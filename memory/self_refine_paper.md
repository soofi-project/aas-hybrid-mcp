# Self-Refine (Madaan et al. 2023) — Paper Notes

## Paper

**"SELF-REFINE: Iterative Refinement with Self-Feedback"** (arXiv:2307.05000)

### Core concept

Single-LLM iterative refinement of free-form text outputs:

1. **Generate**: Initial output `y_0` from input `x`
2. **Feedback**: Same LLM produces specific, actionable feedback `fb_t` on `y_t`
3. **Refine**: Same LLM produces improved output `y_{t+1}` using `x`, all prior outputs and feedback
4. Repeat for up to 4 iterations

Critical: Feedback must be **specific** (identifying concrete phrases to change) and **actionable** (containing a concrete improvement action). Enforced via few-shot exemplars.

### Key results

| Task | GPT-3.5 Base | +Self-Refine | ChatGPT Base | +Self-Refine | GPT-4 Base | +Self-Refine |
|---|---|---|---|---|---|---|
| Sentiment Reversal | 8.8 | 30.4 | 11.4 | 43.2 | 3.8 | 36.2 |
| Dialogue Response | 36.4 | 63.6 | 40.1 | 59.9 | 25.4 | 74.6 |
| Code Optimization | 14.8 | 23.0 | 23.9 | 27.5 | 27.3 | 36.0 |
| Code Readability | 37.4 | 51.3 | 27.7 | 63.1 | 27.4 | 56.2 |
| Math Reasoning | 64.1 | 64.1 | 74.8 | 75.0 | 92.9 | 93.1 |
| Acronym Generation | 41.6 | 56.4 | 27.2 | 37.2 | 30.4 | 56.0 |
| Constrained Gen | 28.0 | 37.0 | 44.0 | 67.0 | 15.0 | 45.0 |

Biggest gains on preference-based tasks (dialogue, constrained gen). Negligible on tasks with clear right/wrong answers (Math: 0% improvement).

### Ablation findings

- **Specific feedback is essential**: Generic feedback ("improve efficiency") provides near-zero benefit. Without feedback, Sentiment Reversal completely fails.
- **Diminishing returns**: First iteration gives the largest improvement; gains taper off by iteration 3.
- **Stronger models benefit more**: GPT-4 sees larger absolute gains than ChatGPT.
- **Self-Refine beats "generate 4 samples"**: Iterative refinement is intrinsically more valuable than sampling multiple times.
- **Weak models fail**: Vicuna-13B cannot reliably produce feedback in the required format. Requires strong instruction-following capabilities.

### Failure analysis (70 samples)

- **94% of Math failures**: Model says "everything looks good" when errors exist. Feedback is the bottleneck.
- **33%**: Feedback inaccurately pinpoints the error location.
- **61%**: Feedback suggests an inappropriate fix.
- Only **6%** of failures: Refiner incorrectly implements good feedback.

The model is better at *incorporating* feedback than at *generating* it.

### Self-Refine vs. Reflexion

| Aspect | Self-Refine | Reflexion (Shinn et al. 2023) |
|---|---|---|
| Domain | Open-ended text/code generation | Embodied planning, tool-use agents |
| Feedback | Self-generated by the same model | Self-reflection from environment failures |
| Memory | All history in-context | External episodic memory bank |
| Scope | Within-episode (single output) | Cross-episode (past mistakes inform future) |
| External signal | None | Environment feedback (success/failure, tool outputs) |
| Tool use | Not applicable | Core to the design (ReAct tool calls) |

### Relevance to our implementation

**Self-Refine is NOT implemented as an agent variant.** There is no `aas-agent:selfrefine` runner. The paper's scope — refining free-form text outputs — is narrower than what our agents do (multi-step tool use across Neo4j/Weaviate/BaSyx).

**Where it COULD apply:** As a post-processing layer on the synthesizer's final answer. After evidence is gathered and combined, Self-Refine could iteratively improve the answer's clarity, completeness, and factual grounding. This would be orthogonal to the existing variants.

**Where it does NOT apply:** The planning/execution/backtracking loop. Reflexion (`reflexion.py`) covers this domain with environment-based feedback, episodic memory, and tool-aware reflection. Self-Refine has no mechanism for tool use or environmental feedback.

### Assessment

Self-Refine is a clean, focused technique for a specific problem (text quality refinement) that is largely orthogonal to our agent orchestration patterns. It's worth mentioning as related work in the paper but does not justify a dedicated agent variant. If a refinement layer is desired, it would most naturally fit inside the CRAG synthesizer (iterative answer refinement) or as an optional post-processing step on all variants.
