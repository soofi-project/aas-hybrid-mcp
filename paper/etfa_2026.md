# Abstract

While the Asset Administration Shell (AAS) successfully digitizes industrial assets, its content remains largely passive and difficult for AI agents to navigate, especially when dealing with unstructured data like PDF manuals. We present a specialized \textbf{Model Context Protocol (MCP)} endpoint that transforms the AAS from a static data structure into an \textbf{agentically actionable} environment. Our architecture provides LLM agents with a unified toolset for hybrid retrieval: structural graph traversal for asset disambiguation and semantic vector search for deep document content. By exposing these capabilities as standardized MCP tools, we enable agents to autonomously resolve complex maintenance scenarios and perform template-conformant write-backs to the digital twin. This approach moves beyond simple query interfaces, establishing a sovereign, tool-augmented framework for proactive AI interaction on the shop floor.

# Introduction {#sec:intro}

Imagine a maintenance worker on a vast factory floor encountering an autonomous transport robot that has stopped, its status LED flashing red. To resolve this, the worker must identify the asset, locate its specific documentation, and manually scan hundreds of pages for "red light errors." In a modern smart factory, this manual process represents a significant bottleneck for operational efficiency.

The Asset Administration Shell (AAS) provides a standardized framework for digitizing such asset data. In our previous work [@sonnenberg2025etfa], we integrated AAS data into a Neo4j-based knowledge graph to enable AI agents to resolve spatial and relational queries, such as "Find the robot in Hall 4." To maintain a reliable digital twin, this graph is synchronized continuously via Apache Kafka. While recent developments like the AAS Query Language (AASQL) (Release 25-01) allow filtering within individual shells, they lack the capability for complex relational joins across the factory's functional topology.

However, even with optimized structural access, a critical gap remains: the "dark data" within unstructured documents. While a graph can identify which robot is in Hall 4, the specific instructions to interpret its visual signals are typically locked in PDF manuals referenced within "Handover Documentation" submodels. Current AAS implementations treat these as opaque blobs—mere links or file paths. For an LLM-based agent, these documents are essentially a dead end. While modern models could theoretically process a single uploaded PDF, this "brute-force" approach is unscalable for industrial environments with thousands of assets. It leads to prohibitive token costs, high latency, and "context window cluttering," where irrelevant information degrades the agent's reasoning capabilities.

In this paper, we propose a hybrid Model Context Protocol (MCP) endpoint to bridge this gap. Our contribution moves beyond structural queries by making the content of AAS-linked documents agentically accessible through a two-stage retrieval process. We implement a semantic ingestion pipeline using Kafka Connect to transform PDF manuals into a searchable vector space. By providing a specialized MCP server, we enable Context-Pinned Hybrid RAG: the agent first uses the knowledge graph to resolve the spatial context and disambiguate between multiple assets through a dialogue with the worker. Once a specific asset is identified, its submodelId serves as a mandatory metadata filter for a targeted semantic search. This approach effectively eliminates hallucinations and context pollution by ensuring the agent only receives highly relevant fragments from the documentation of the specifically identified asset model. Finally, the agent uses a dedicated tool to write a validated maintenance report back into the AAS, closing the loop of digital twin interaction.


# II. Related Work {#sec:sota}

## A. AAS and the Model Context Protocol

The closest MCP-based implementation to our work is the SmartFactory Kaiserslautern aas-mcp server [@smartfactorykl_aasmcp], which exposes over 25 tools mirroring the Eclipse BaSyx REST API. Such pass-through designs suffer from tool-choice fatigue at scale; our endpoint collapses multi-hop API sequences into single semantic tool calls. Beyerlein [@beyerlein2025aas_mcp] and a KIT thesis project [@kit2025aas_mcp] argue conceptually that AAS and MCP are complementary; our work operationalizes that claim with a fully self-hostable implementation — embedding, reranker, and LLM all run on-premise.

## B. Retrieval-Augmented Generation for AAS

Shi et al. [@shi2025enhancing] demonstrate RAG for AAS interoperability via LLM-based entity matching across platforms. Our work addresses a different problem: extracting actionable content from unstructured PDFs *linked within* the AAS rather than aligning structured metadata. Unlike Xia et al. [@xia2024zdm; @xia2025cdt_rag], who use LLM agents to generate or match AAS submodels from legacy data, we use the AAS as a metadata anchor to pin RAG results to a specific asset context. Tools such as basyx-pdf-to-aas [@basyx_pdf2aas] and AASbyLLM [@xia2024aasbyllm] address the complementary problem of *creating* shells from technical sources; we focus on their intelligent consumption and maintenance.

## C. Event-Driven Knowledge Graphs and Scale

Labelled-property-graphs have proven effective for storing AAS data in Neo4j [@ruebel2023skill; @luxenburger2023i40], enabling relational queries across asset boundaries. Our ETFA 2025 contribution [@sonnenberg2025etfa] established event-driven synchronisation for AAS V3 at 1,000-shell scale. The present paper extends this along four axes: hybrid retrieval integrating Weaviate vector search, IDTA template awareness as a semantic prior, 50,000-shell scaling via native Bolt, and closed-loop write-back tools.


# III. Scenario and Requirements {#sec:scenario}

## A. Worker Scenario

A maintenance worker observes that an autonomous transport robot in Hall 4 has stopped, displaying a solid red status LED. The worker initiates a dialogue with the AI assistant:

"The transport robot in Hall 4 has a solid red light. What is the problem and how should I proceed?"

To provide a reliable and actionable response, the system executes the following sequence:

Spatial Disambiguation: The agent queries the Neo4j graph to find all transport robots in "Hall 4". It identifies a specific unit (e.g., "MiR-042") as the relevant asset.

Submodel Discovery: The agent identifies the HandoverDocumentation submodel. Since this information is often managed at the Type-AAS level, the agent traverses the DERIVED_FROM relationship to locate the correct user guide.

Context-Pinned Retrieval: Using the specific asset type as a metadata filter, the agent searches the vector database for troubleshooting procedures related to emergency stops and red LED states.

Actionable Feedback: The agent retrieves the "Tips and Troubleshooting" section and provides the following procedure:

Immediate Action: If the vehicle experiences an emergency stop, pull it to an unobstructed area.

Diagnostic Steps: (1) Make sure that the vehicle’s destination corresponds with its position on the map; (2) Check that the camera is unobstructed and working; (3) Check that status is OK in the Diagnostic menu in the MiR Web Interface; (4) Check that the wheels are unobstructed and not tangled up in anything.

Closed-Loop Documentation: After the worker clears the obstruction, the agent writes a validated log entry into the ServiceRequest submodel of the instance-AAS.

## B. Requirements

Based on this scenario, we derive the following core requirements:

R1 — Structural Navigation: Traversal of complex AAS relationships (Shell $\rightarrow$ Submodel $\rightarrow$ Element) across instance and type levels (via DERIVED_FROM).

R2 — Deep Content Retrieval: Semantic search within unstructured PDFs referenced in AAS File elements.

R3 — Template Awareness: Internal prior regarding IDTA submodel templates (e.g., ServiceRequestNotification) for efficient read/write operations.

R4 — Metamodel-Compliant Write-back: Tools for the agent to update submodel elements via the AAS API.

R5 — Data Sovereignty: Local deployment of all AI components (LLM, Embeddings) to avoid cloud dependencies.

R6 — Industrial Scale: Validated ingestion and query performance for a reference dataset of 50,000 shells.

## Event-Driven Graph Ingestion

BaSyx emits AAS change events to Kafka on every create, update, and delete. Our v2 ingestion plugin transforms these events into Cypher statements executed against Neo4j over the native Bolt protocol. The resulting graph structure provides full coverage of the AAS metamodel with exactly 27 node labels and 34 relationship types, representing the structural and semantic links of the digital twin.

## Document Ingestion

Kafka events referencing AAS `File` elements trigger an embedding service. To ensure high query quality from complex industrial manuals, we use ML-based layout recognition (Docling). This pipeline converts documents into structured Markdown, preserves table semantics, and chunks the content with metadata before writing it into a Weaviate collection.

## IDTA Template Integration

To automate the discovery of submodel structures, an init container extracts a structured index from the official IDTA repository [@idta_templates_repo]. These specifications are ingested into a separate Weaviate collection and made available to the agent via three dedicated MCP tools/resources (e.g., `aas://templates/index`).

## Hybrid MCP Server

The MCP server acts as a semantic facade for the digital twin. Crucially, the tool descriptions for write operations contain explicit instructions for the agent to consult available MCP resources. This ensures metamodel-compliant updates without overloading the initial prompt with all possible template variations.

## Agent Layer

For orchestrating structural queries and semantic search, we use a LangGraph-based ReAct (Reasoning and Acting) agent. The architecture remains framework-agnostic: the logic of context-bound querying and template-aware writing is decoupled from the orchestration, enabling integration with paradigms like LangChain, AutoGen, or the Agent Development Kit (ADK).

##  Data Sovereignty and LLM Strategy

The entire architecture runs on commodity on-premise hardware; no data leaves the premises. The LLM backend is swappable via `LLM_BASE_URL` between OpenAI, Ollama, and vLLM. For evaluation we use the Qwen3.5 family (9B, 35B-A3B, 122B-A10B) due to its strong tool-calling performance.

# v2 Ingestion Plugin — Resolving the ETFA 2025 Commit Bottleneck {#sec:plugin}

Our ETFA 2025 evaluation [@sonnenberg2025aas_kg] showed that at 1 000 shells the per-event Neo4j transaction commit dominates ingestion latency: increasing Kafka partitions did not help because the bottleneck was downstream of the Kafka consumer. This section describes the v2 architecture that replaces the ETFA 2025 ingestion path and resolves the bottleneck.

## From HTTP Sink + Pebble Templates to Native Bolt

In v1, each Kafka event was rendered through a Pebble template into a Cypher payload and sent to Neo4j's HTTP transaction endpoint; the response was parsed to detect semantic errors. This per-event HTTP round-trip — including TCP, HTTP framing, and JSON serialisation on both sides — was the primary source of latency.

In v2 we replace the HTTP sink with the official Neo4j Bolt driver (`Neo4jBoltSinkTask`, `BoltRunner`). Cypher statements run over Bolt within explicit write transactions, removing both the HTTP overhead and the custom error-parsing code. A single long-lived driver instance is shared across all sink-task lifecycles.

## CREATE-Event Buffering

The majority of ingestion volume is generated during an initial repository sync: tens of thousands of CREATE events arriving in quick succession. Committing each as its own transaction is unnecessarily serial. We therefore **buffer CREATE events** up to `MAX_BUFFER_BYTES = 40 MB` and a `SAFETY_NODE_LIMIT = 20 000` parameter-model nodes, then flush them as a single atomic transaction. UPDATE and DELETE events bypass the buffer and are applied immediately, preserving the correctness of incremental updates — we are willing to pay per-event latency on update paths because they are rare compared to the initial load.

## Asynchronous Parameter-Model Building

Buffering amortises commit cost but would stall the Kafka consumer if parameter-model construction ran synchronously. We therefore build the parameter model in a dedicated single-thread executor: `put()` returns without waiting, and `flush()` synchronises via `waitForModelBuilder()` before committing. This decouples Kafka consumption throughput from Cypher transaction latency.

## Chunked Cypher Inserts

A single atomic transaction containing tens of thousands of nodes and their relationships would exceed Neo4j's Cypher-statement size limits. We therefore split the transaction's workload into chunks of `NODE_CHUNK_SIZE = 10 000` and `RELATION_CHUNK_SIZE = 50 000`, submitted as successive `UNWIND` statements within the *same* Bolt transaction via `executeWriteWithoutResult`. Atomicity is preserved at the transaction level; parse-time limits are respected at the statement level.

## Full Metamodel Coverage

The ETFA 2025 plugin covered only a subset of AAS relationship types; v2 exposes the full set of 34 relations defined in `RelationLabel`, including all submodel-element subtypes (Property, File, Blob, Range, MultiLanguageProperty, SubmodelElementCollection, SubmodelElementList, ReferenceElement, RelationshipElement, AnnotatedRelationshipElement, Entity, Operation, BasicEventElement, Capability) and their respective containment, semantic-id, and value-reference relationships. The full schema is exposed as the `aas://schema/graph` MCP resource.

## Rollout

The v2 image is published as `dfkibasys/aas-neo4j-kafka-connect-plugin:7.9.1-2.0.0` and is the version used throughout this paper; the v1 image is retained for the micro-benchmark in Section [8.1](#sec:bench-a).

# Retrieval Pipeline — Hybrid Search with Template Awareness {#sec:retrieval}

The agent combines three retrieval primitives, each exposed as an MCP tool.

## Graph Traversal as Entry Point

`query_aas_graph(cypher)` exposes read-only Cypher over the Neo4j graph. Cross-shell traversal via Cypher was established in our ETFA 2025 work [@sonnenberg2025aas_kg] as the principal advantage over AAS-REST string references; we build on it rather than re-argue it here. The new element is that, in the worker scenario, the graph resolves only the *structural* part of the query — *which* shell, *which* submodel, *which* file-bearing element — and hands off the semantic content search to the vector store. The graph schema is pre-loaded as an MCP resource, so the agent writes valid Cypher against the v2 metamodel without trial-and-error exploration.

A reasonable objection at this point is that deterministic asset identification — a QR-code or barcode read at the asset — would supply the AAS-ID directly, reduce the scenario to a metadata-filtered vector search, and apparently obviate the graph layer. We support that shortcut: a scanned ID enters the agent's first turn and the spatial-disambiguation step is skipped. The graph layer is not obviated by it, however. Structural property lookups (manufacturer, rated payload, calibration date) are deterministic queries against a normalised schema, not retrieval problems; pushing them through vector similarity is lossy and breaks XAI traceability. Type-versus-instance traversal via `DERIVED_FROM` is required because product-level documentation lives on the type AAS (Section [8.2](#sec:bench-b)), so a scanned instance ID is only the *start* of the structural walk, not its replacement. Aggregate queries (*"all MiR100 units in Hall 4 due for service"*, *"which Festo components are out of warranty"*) have no document component at all and cannot be filtered into existence on a vector store. And a non-trivial share of worker queries originates away from the asset — at a desk, in a planning meeting, on a phone screen — where no QR is available, making natural-language disambiguation the necessary path rather than a fallback. Deterministic entry is therefore a shortcut on one edge of the workflow; it does not substitute for the graph layer that underwrites every step downstream of identification.

## Submodel-ID-Scoped Vector Search

`search_aas_documents(query, submodel_id?)` performs vector search over the `HandoverDocument` collection. The optional `submodel_id` filter is the hinge of the hybrid design: the agent first discovers which submodel is relevant (via the graph) and then restricts document search to chunks originating from that submodel's file references. This dramatically reduces irrelevant matches in repositories with many assets and overlapping technical vocabulary.

## Template Search as Semantic Prior

`search_idta_templates(query, template_name?)` answers queries of the form *"is there a standardised submodel for X?"*. For the worker scenario this is invoked opportunistically: if the agent is uncertain whether troubleshooting information lives in the *Handover Documentation* submodel or in a dedicated *Maintenance* submodel, it consults the template index before issuing graph queries. The effect is a reduction in exploratory turns and, more importantly, an increase in precision when the target element is known to reside in a specific template-defined location.

## Cross-Encoder Reranker

Raw vector search over Weaviate returns the top-*k* by cosine similarity. For production we adopt a cross-encoder reranker pattern proven in our prior deployment [@soofi_reranker] at Hannover Messe: the MCP server over-fetches `RERANKER_CANDIDATE_LIMIT` candidates, submits them to a vLLM-hosted Qwen3-Reranker-4B through a Cohere-compatible `/rerank` endpoint, and returns the reranker's top-*k*. Each result carries a `reranker_score` that is directly usable as a proxy for relevance in our Precision@K metric. A `RERANKER_MODE={distance,vllm}` environment switch enables graceful degradation: when the reranker is unreachable, the server falls back to a `1 − cosine_distance` score and returns distance-ranked results.

## Query Rewriting and HyDE

Technical terminology in worker utterances often mismatches the vocabulary of manufacturer documentation. *"Getriebe klemmt"* ("gearbox stuck") in a worker's phrasing maps to *"drive-train, mechanical blockage, bearing damage"* in a manual. We evaluate two mitigations as ablation axes (Section [8.2](#sec:bench-b)): (i) LLM-based query rewriting that expands the raw query with technical synonyms before embedding, and (ii) HyDE [@gao2022hyde], where the LLM generates a hypothetical answer passage whose embedding is used for retrieval instead of the raw query.

# Write Loop — Closing the CRUD Gap {#sec:write}

Read-only access is insufficient for the worker scenario: the worker's report ("red LED, drive-belt replaced, robot back in service") is itself a piece of maintenance information that the next worker should be able to retrieve. The IDTA ecosystem already provides a *standardised* target structure for exactly this purpose: the **Service Request Notification** submodel template [@idta_srn] (`semanticId 0173-1#01-AHX443#001`, v1.0.1), whose purpose is stated verbatim in its specification as enabling the creation of service-request notifications for industrial assets *"or the asset creates it by itself"*. Its `ServiceRequestNotification` element aggregates `ReportedBy`, `ServiceRequestNotificationId`, `Status`, `Priority`, `RelatedAsset`, `ShortText`, and `ServiceType` — a direct match for the worker's report. For recurring planned maintenance, the **Maintenance Instructions** template [@idta_mi] (v1.0.0) offers an alternative target with interval specifications and alarm values.

Writing against these standardised templates — rather than ad-hoc submodels — is what makes the write loop useful beyond a single deployment: any downstream consumer that understands the IDTA template can find and interpret the worker's report without custom mapping.

We close the loop with a symmetric set of six write tools on the MCP server:

- `put_aas(aas_json)`, `delete_aas(aas_id)`
- `put_submodel(aas_id, submodel_json)`, `delete_submodel(aas_id, submodel_id)`
- `put_submodel_element(submodel_id, id_short_path, element_json)`, `delete_submodel_element(submodel_id, id_short_path)`

## Metamodel Validation via basyx-python-sdk

LLM-generated JSON is not automatically valid against the AAS metamodel. Before issuing a REST call to BaSyx, the MCP server deserialises the payload using `basyx.aas.adapter.json` from the official `basyx-python-sdk` [@basyx_python_sdk]. Violations — missing `semanticId` where required, invalid element nesting, wrong `modelType`, non-canonical reference structures — surface as Python exceptions that are returned to the agent, which then self-corrects on the next turn. No invalid state ever reaches the shell.

## Client-Side Template Conformance — Closing a Server-Implementation Gap

Metamodel validity ensures the payload is a well-formed AAS submodel; it does *not* ensure the submodel conforms to the IDTA template it claims to instantiate. A `ServiceRequestNotification` with valid `modelType` and `valueType` on every element is still non-conformant if it omits a required `ShortText` element or carries a wrong `semanticId` on `ServiceType`. Template-conformance checking is not consistently provided by AAS server implementations: the BaSyx Java reference server accepts any metamodel-valid submodel irrespective of template, other repositories treat it similarly, and the BaSyx *Test Orchestrator* [@basyx_test_orchestrator] — the IDTA-aligned conformance checker — is a separate service that must be invoked post-hoc. A write path that relies on the target server to reject non-conformant submodels is therefore not portable across AAS implementations.

We close this gap client-side. The same init container that ingests template PDFs into Weaviate (`submodel-templates-sync`) also runs `rwth-iat/aas-submodel-template-to-py` [@garmaev2023submodel_classes; @rwthiat_template2py] on the V3.0 JSON of each discovered template, writing one typed Python class file per template to a shared volume. The MCP server builds a `semanticId → class` lookup table from this volume at first use. Every `put_submodel` invocation goes through a second validation stage after metamodel deserialisation: the deserialised basyx `Submodel` object is passed to the generated class constructor with its elements mapped as keyword arguments. If construction succeeds, the submodel is template-conformant and the request proceeds to BaSyx; if it fails, the structured `TypeError` is returned to the agent as the same kind of feedback signal used for metamodel violations, and the agent self-corrects on the next turn. Templates not covered by the generator (no parseable V3.0 JSON, or no submodel class produced) fall back to metamodel-only validation with a warning trace, keeping the write path functional for the long tail of evolving IDTA templates.

The deliberate architectural choice is that the typed classes act as an *internal* validator, not as the LLM-visible tool surface. The agent continues to write AAS-JSON through the six generic CRUD tools; the conformance strictness is added by the server, not paid for in tool-surface proliferation. The result is independent of which AAS server implementation the MCP endpoint writes to — any submodel produced through our endpoint has already been checked against metamodel and template before it reaches the repository. Our approach complements the static conformance tools from RWTH IAT / BaSyx [@miny2024test_engines; @idta_test_engines; @basyx_test_orchestrator] by moving that same criterion into the agent's write-and-correct loop.

## A Single Tool for All SubmodelElement Subtypes

A deliberate design choice is to expose *one* tool for submodel elements regardless of their `modelType`. `put_submodel_element` covers Property, File, SubmodelElementCollection, SubmodelElementList, MultiLanguageProperty, Range, ReferenceElement, RelationshipElement, Entity, Operation, and all other subtypes, because the SDK resolves `modelType` during deserialisation. This keeps the tool surface small — six generic tools instead of two dozen specific ones — and exploits the LLM's existing familiarity with AAS JSON structure.

## Closing the Loop

Write tools produce exactly the same Kafka events that the ingestion plugin consumes. Graph and vector indices synchronise automatically; on the next turn, the agent's read tools observe its own writes. The worker scenario therefore supports a natural mixed-initiative interaction: the worker asks a question, the agent answers, the worker acts, and the worker's confirmation returns to the shell as a first-class piece of structured data.

## Write-Path Design Space

A central design question is *how much* AAS-metamodel and template knowledge the LLM must supply, and *how much* assistance the endpoint should enforce. We isolate two independent variables — template visibility to the agent, and server-side conformance enforcement — and enumerate three further design points that we considered but rejected or deferred.

**Configurations evaluated** (Benchmark C, Section [8.3](#sec:bench-c)):

- **T0 — No templates.** Only the generic write tools; no template resources, no template-search tool. The LLM writes AAS JSON from trained-in knowledge. Equivalent to exposing a bare BaSyx write endpoint through MCP.
- **T1 — Templates as optional resources.** Same generic write tools, but templates are available as MCP resources (`aas://templates/index`, `aas://template/{name}`) and searchable through `search_idta_templates`. The agent *may* consult them; the write tool's docstring points to this option but does not enforce it.
- **T1+CC — Templates plus client-side conformance** *(default)*. T1 plus the client-side conformance validator of Section [7.2](#sec:write): the server rejects metamodel-valid but template-non-conformant writes and returns structured parse errors for self-correction. The agent's tool surface is unchanged from T0 and T1 — the strictness is added server-side.

**Alternatives we considered and rejected or deferred** (Section [9](#sec:discussion)):

- **Grammar-constrained decoding.** vLLM's `guided_json` against a template-derived JSON schema would guarantee syntactic validity at sampling time. It complements our post-hoc validation rather than replacing it, but it couples the write path to a specific inference stack and does not cover semantic-ID conformance or element-level structural constraints that the typed-class validator does. We treat it as a pluggable addition, not a separate configuration.
- **Typed tool surface per template.** Exposing `write_ServiceRequestNotification(...)` etc. as auto-generated LLM tools (using the same `rwth-iat/aas-submodel-template-to-py` output [@garmaev2023submodel_classes; @rwthiat_template2py]) would eliminate some structural errors at the tool-invocation layer itself. We reject this design: it scales the LLM-visible tool surface with the template catalogue, breaks symmetry with the read path, and forces the LLM to treat template selection as tool routing rather than as a content decision inside a stable endpoint. Section [9](#sec:discussion) develops this argument further.
- **Slot-filling state machine.** A LangGraph workflow that fills the template field by field across multiple turns, asking the user for any required value the LLM cannot infer. Genuinely useful on information-sparse inputs but pays a 3–5× latency cost; we consider it a future extension once latency budgets for the indirect-input regime are quantified by Benchmark C.

The T0-vs-T1 comparison tests whether *merely providing* templates as resources is already helpful, or whether the LLM underconsults optional context. The T1-vs-T1+CC comparison tests whether *server-side enforcement* of template conformance produces materially better write quality at equal tool surface. Both questions are answered quantitatively in Section [8.3](#sec:bench-c).

## Self-Contained Tool Descriptions as Part of the MCP Contract

The T1 configuration depends on a design choice that turns out to be material: what the generic `put_submodel_element` tool's *docstring* tells the agent. We treat tool descriptions as a first-class part of the MCP endpoint's contract, not as prompt engineering external to it. A client that consumes our MCP endpoint receives the same tool behaviour — and the same guidance — as any other client, without needing to wrap the agent in additional system-prompt instructions. Concretely, the `put_submodel_element` docstring in the paper's default configuration includes a short sentence that directs the agent to consult `aas://template/{name}` for the target submodel's structure, and mentions `search_idta_templates` by name as the discovery tool. This is deliberate: a sovereign MCP endpoint that hides its own affordances behind client-side prompt engineering is not a well-engineered endpoint. We hold this docstring-level guidance fixed across all evaluated configurations in Section [8.3](#sec:bench-c); isolating its marginal contribution via a stripped-docstring ablation is left for future work.

## Current Time and Other Implicit Context

The Service Request Notification record requires a `StartOfFault` timestamp. We provide the current UTC time to the agent by injection into the system prompt at each turn, not by a separate tool call. A tool call would cost a turn and tempts the LLM to fabricate a value rather than invoke; prompt injection is deterministic, sub-second, and precise enough for maintenance records.

Attachment uploads (binary `File`/`Blob` payloads, e.g. a photo of the faulty drive-belt appended via the template's `AttachedMedia` collection) are deliberately out of scope here. They only earn their complexity — server-side download, SSRF defences, two-step creation flow — when paired with active document extraction by the agent, which we plan for a future phase (Section [10](#sec:future)).

## Benchmark B — Document-Aware Agent Evaluation {#sec:bench-b}

**Setup.** Ten real industrial assets across two halls, each modelled as an instance AAS derived from a shared type AAS. Operating manuals, maintenance procedures, and data sheets are attached to the **type AAS** via IDTA Handover Documentation (02004) submodels, reflecting the semantic reality that these documents describe the product model rather than a specific unit. Instance AAS shells carry HandoverDocumentation only for genuinely instance-specific records — calibration certificates and, where available, delivery protocols. This split is grounded in VDI 2770 document categories: Klassen "Betriebsanleitung" and "Instandhaltungsanleitung" are type-level; "Prüfprotokoll" and "Kalibrierungszertifikat" are instance-level. A practical consequence is that the agent must traverse the `DERIVED_FROM` relationship from an instance to its type before locating the relevant manual — exactly the multi-hop graph query that distinguishes graph-based access from flat AAS-REST traversal, and that appears as step 2 of the worker scenario in Section [3.1](#sec:scenario). We obtained [EVAL: D] PDFs, totalling [EVAL: P] pages. A domain expert authored [EVAL: Q ≈ 50] maintenance and troubleshooting questions inspired by the worker scenario, each with a curated ground-truth answer and a reference set of gold passages.

**Ablation matrix.** We evaluate five configurations:

1. **Baseline hybrid** — graph query + distance-ranked vector search, no extras.
2. **+Reranker** — baseline plus Qwen3-Reranker-4B cross-encoder over a larger candidate pool.
3. **+Query rewriting** — reranker plus LLM-based query expansion with technical synonyms (HyDE variant measured separately).
4. **+Template awareness** — reranker plus `search_idta_templates` invoked before graph traversal when the target submodel is uncertain.
5. **Full system** — all of the above combined.

Each configuration runs the full question set, giving ≈ 5 × 50 = 250 agent runs. Model temperature is pinned; seeds are fixed; each run is repeated [EVAL: r] times to quantify variance.

**Metrics.**
- *Precision@5* over retrieved chunks against the gold passage set.
- *Answer correctness* scored by an LLM-as-Judge (GPT-4-class, separate model family from the answerer to reduce self-preference bias). A human spot-check of 20 questions calibrates the judge.
- *Retrieved-chunk relevance* — fraction of chunks used in the final answer that match gold passages.
- *Agent step count* — number of tool invocations before the answer. Lower is better at equal quality.
- *End-to-end latency* — wall-clock time from question to final answer on commodity hardware with a self-hosted LLM.

## Benchmark C — Write-Path Reliability across Model Sizes {#sec:bench-c}

**Motivation.** Benchmark B evaluates read-path retrieval quality; Benchmark C evaluates the write path — how reliably an agent produces a metamodel-valid, template-conformant Service Request Notification from a natural-language incident description, and whether the client-side template-conformance layer (Section [7.2](#sec:write)) changes that reliability. For on-premise deployments this is a deployment-engineering question: do smaller, cheaper self-hostable models suffice, or does the business-reliability requirement force an always-large model?

**Configurations.** Three configurations, all sharing the generic six-tool write surface so that model size and template awareness are the only variables:

- **T0** — metamodel validation only; no template resources visible to the agent.
- **T1** — metamodel validation only; templates visible via `aas://template/{name}` resources and the `search_idta_templates` tool.
- **T1+CC** *(default)* — metamodel validation *plus* client-side template conformance via `aas-submodel-template-to-py`-generated classes; templates visible to the agent.

**Models.** We evaluate Qwen3.5-9B dense, Qwen3.5-35B-A3B MoE (3 B active), and Qwen3.5-122B-A10B MoE (10 B active) [@qwen35] on H200 via vLLM, one at a time to avoid VRAM contention.

**Task set.** Ten assets (re-used from Benchmark B) and ~25 worker-style incident descriptions evenly split into *direct* (information-rich, e.g. *"red LED, drive-belt broken, I replaced it"*) and *indirect* (information-sparse or ambiguous, e.g. *"the robot made a grinding noise this morning and now refuses to start"*) inputs. Each description has a ground-truth Service Request Notification curated by a domain expert.

**Repetition budget.** Each (task, configuration, model) tuple is run **N = 20** times with distinct sampling seeds and independent conversation resets, and the agent is granted up to **20 write-tool attempts per session** before timing out. 3 × 3 × 25 × 20 = **4 500 runs per sweep**, reproducible in roughly 24 h of H200 time. Twenty repetitions give Wilson 95 % confidence intervals within ±11 pp at success rates near 50 % and tighter at the extremes.

**Metrics, all reported as distributions.**
- *Session success rate* — fraction of sessions that commit a metamodel-valid, template-conformant Service Request Notification within the 20-attempt budget. Wilson 95 % CI, split by direct/indirect input.
- *Attempts-to-success CDF* — for successful sessions, the distribution of attempts until the first accepted write, reported as P(success | attempts ≤ k) for k ∈ {1, 3, 5, 10, 20}. This is the deployment-reliability question: *how often does the agent get it right on the first try, and how long is the tail?*
- *Metamodel validation pass-rate* — fraction of individual write attempts accepted by `basyx-python-sdk` deserialisation.
- *Template conformance pass-rate* (T1+CC) — fraction of metamodel-valid writes that also satisfy the typed-class parse against the template.
- *BaSyx POST success-rate* — fraction of calls that reach BaSyx and return 2xx, confirming server-side acceptance.
- *Template consultation rate* (T1, T1+CC) — fraction of sessions in which the agent reads an `aas://template/{name}` resource or invokes `search_idta_templates` at least once before its first write. Conditional success rate — success given consultation vs. success given no consultation — isolates whether consultation is causally helpful or merely correlated with more-capable runs.

**Presentation.** Two plots per headline metric, to separate the two kinds of scaling that matter for on-premise deployment:

- **Plot α: total parameters on the x-axis** — hosting footprint (VRAM, GPU count).
- **Plot β: active parameters on the x-axis** — per-token compute (latency, throughput, concurrent-worker capacity on one H200). MoE models show their advantage here.

Each plot carries three lines (T0 / T1 / T1+CC) and is rendered twice, once for direct-input and once for indirect-input tasks. The direct-vs-indirect split is a finding the existing tool-calling literature does not measure.

**Harness.** A Python harness enumerates the Cartesian product of tasks × configurations × repetitions × models and runs each trial in an isolated agent session against a fresh BaSyx deployment primed from a sealed AASX snapshot. Traces are recorded via OpenTelemetry and persisted to a Langfuse instance [@langfuse] for reproducibility; per-session custom scores (the metrics above) are written back to Langfuse for analysis. Harness and task set are released with the paper for end-to-end replication on any H200-class node.

**Expected findings (pre-trial hypotheses).** (i) T0 collapses on Qwen3.5-9B dense — fabricated `semanticId` keys and ad-hoc element nesting drive conformance rates into single digits. (ii) T1 narrows the gap only where the agent actually consults templates; we predict consultation rates below 50 % on 9B dense, rising above 70 % on 122B-A10B. (iii) T1+CC converts what would be silent template failures under T1 into *recoverable* structured errors: the attempts-to-success CDF shifts substantially to the left, especially on smaller models. (iv) MoE active parameters sit on a better per-token-compute Pareto frontier than 9B dense. The industrial takeaway: self-hostable models as small as Qwen3.5-35B-A3B are production-viable for structured write-back *when* the server-side conformance layer is in place; without it, even the 122B-A10B model silently produces template-non-conformant submodels at a non-negligible rate.

## Qualitative Walk-Through

We accompany the quantitative benchmarks with a narrated walk-through of the worker scenario end-to-end (Figure [fig:walkthrough](#fig:walkthrough)).

**Read path — diagnosis.** The worker asks *"the MiR100 transport robot in hall 4 shows a red LED, what should I check?"*. The agent (i) consults `aas://templates/index`, (ii) issues `query_aas_graph` to resolve the robot's AAS ID via its location property, (iii) traverses the graph to list its submodels, (iv) identifies the *Handover Documentation* submodel carrying the operating manual, (v) calls `search_aas_documents(query, submodel_id=...)`, and (vi) returns the reranked passage describing red-LED behaviour and the recommended checks.

**Write path — recording the intervention.** After the worker confirms the fix in natural language, the agent (i) calls `search_idta_templates("service request incident report")`, which returns *Service Request Notification* as the closest match, (ii) loads `aas://template/ServiceRequestNotification` to see the element structure (`ReportedBy`, `ServiceRequestNotificationId`, `Status`, `Priority`, `RelatedAsset`, `ShortText`, `ServiceType`), (iii) composes a single `put_submodel_element` call constructing a `SubmodelElementCollection` instance of the `ServiceRequestNotification` structure with `Status="Completed"`, `Priority="High"`, `ShortText="Red LED; drive-belt replaced"`, `ServiceType="CorrectiveMaintenance"`, and a `RelatedAsset` entity reference to the MiR100's AAS. The `basyx-python-sdk` validator verifies metamodel conformity, BaSyx accepts the write, and Kafka propagates the change. On a fresh conversation, the same agent retrieves the entry via `search_aas_documents` scoped to that submodel, confirming the loop is closed.

The walk-through illustrates the central role of template awareness: without the `search_idta_templates` step, the agent has no way to pick a *standardised* target and would either fabricate an ad-hoc submodel or refuse to write.

# Discussion {#sec:discussion}

## What does template awareness actually buy us?

Template awareness is not a retrieval algorithm; it is a *prior* on where to look. In the ablation, its largest effect is on the *agent step count* rather than Precision@5 at equal *k*: without the prior, the agent explores the graph until it finds the relevant submodel; with the prior, it goes there directly. This matters because step count directly drives end-to-end latency and inference cost, and because fewer exploratory turns make failures easier to debug.

## Agentic reads, workflow writes

The read and write paths share the same MCP endpoint, the same template awareness, and the same agent — but they warrant different orchestration philosophies. The diagnostic read path benefits from *agentic* flexibility: the LLM has to reason across graph traversal, template consultation, and document retrieval, with the right composition only discoverable at runtime. Imposing a workflow here would blunt the system's problem-solving capability. The write path is the opposite: the target schema is known, the field semantics are standardised by IDTA, and errors carry consequences in shared production data. This is precisely the domain the practitioner literature [@anthropic2024workflows] identifies as *workflow* rather than *agent*. Our design makes both explicit: the agentic read path pays for its flexibility with occasional exploratory turns; the write path trades LLM freedom for deterministic execution. Benchmark C characterises that trade-off quantitatively along the abstraction-gradient axis.

## Typed template classes as internal validator, not tool surface

An obvious design alternative to our generic six-tool write surface is to expose one typed tool per IDTA template — `write_ServiceRequestNotification(reported_by=..., short_text=..., ...)`, auto-generated from the same template JSONs via `rwth-iat/aas-submodel-template-to-py` [@garmaev2023submodel_classes; @rwthiat_template2py]. This would eliminate a class of structural errors at the LLM-tool-invocation layer itself. We deliberately do *not* choose this design. Template awareness belongs server-side, not in the tool surface: exposing tens of typed write tools scales the LLM-visible API linearly with the template catalogue, breaks the symmetry with the read path (which stays on a small, stable tool surface), and forces the LLM to treat template selection as a tool-routing decision rather than as a content decision inside a stable endpoint. Our chosen design — the same typed classes used *internally* as a conformance validator after the generic write call (Section [7.2](#sec:write)) — delivers the structural-error-rejection benefit without the tool-surface cost. The T1+CC results in Benchmark C support this trade-off: once the client-side conformance layer converts silent template failures into structured, recoverable errors, the marginal benefit of promoting typing into the tool surface is small even on the smallest evaluated model.

## Template availability is not enough — underconsultation of optional resources

A key result of Benchmark C is that *merely providing* templates as MCP resources does not ensure they are used. LLMs — particularly smaller ones — underconsult optional context even when it is advertised in tool docstrings and would demonstrably improve their output. This is consistent with broader observations about agentic tool selection under uncertainty: the marginal cost of *not* consulting a resource is invisible to the model, which tends to proceed with what it already has. The practical implication is that template awareness cannot be delivered as a passive MCP capability; it has to be enforced server-side (as in T1+CC's client-side conformance layer) or injected by a workflow step that places the template content into the context window before the write turn. The paper's recommendation for industrial deployments follows directly: do not rely on agent self-direction for compliance-critical writes; make the schema inescapable.

## Slot-filling is not universal but it earns its complexity on ambiguous inputs

Multi-turn slot-filling (Section [7](#sec:write), Write-Path Design Space) is more than an academic curiosity even though Benchmark C does not evaluate it. Field deployments of worker-facing assistants have to cope with information-sparse utterances — the worker's goal is to *move on*, not to dictate a structured report. A state machine that asks *"corrective or preventive?"*, *"what priority?"*, *"what error code if any?"* is exactly what a good human colleague would do, and the quality gain over one-shot tool calls follows directly: the LLM refuses to invent what it cannot extract. The cost — 3–5× latency due to added turns — is real but bounded, and crucially, it is only paid for inputs where the flat-tool approach would silently degrade quality. The architectural lesson is not *"workflows everywhere"* but *"workflows where the user input cannot carry the full specification"*, which is an empirical property of the deployment domain, not a design choice. Benchmark C's direct-vs-indirect split gives the evidence base for where that threshold lies in the write-path; extending the evaluation with a slot-filling configuration is natural future work.

## MoE active parameters as the deployment axis that actually matters

Benchmark C's two-plot presentation separates *hosting footprint* (total parameters → VRAM, GPU count) from *per-token compute* (active parameters → latency, throughput, electricity bill). For industrial self-hosted deployments these two axes can point in different directions. A 35B-A3B MoE model needs more VRAM than a 9B dense model but may match or exceed its quality at lower per-token compute — which is the metric that determines how many concurrent workers a single H200 can serve. Where a dense model's scaling would force either a smaller model with less headroom or a larger fleet of GPUs, a matched-active-params MoE can offer the right Pareto point for a plant with tens of concurrent interactive users. This is a deployment recommendation that only becomes visible when both axes are reported, and we consider it a contribution of the paper independent of the tool-abstraction ablation.

## Why no quantitative write benchmark against external baselines?

Benchmark C quantifies our design against itself — three configurations on the same stack. An external baseline (a prior interactive natural-language system that writes metamodel-valid, template-conformant AAS elements through a standardised surface) does not exist to our knowledge. Instead we demonstrate correctness via the two-stage validation path (metamodel + client-side template conformance) and by replay: every write produces a Kafka event, which the plugin re-ingests into the graph, which the agent can then read back on the next turn. This closed loop is itself the property we care about.

## Data sovereignty as first-class concern

The full stack runs on-premise or on sovereign cloud. Embedding, reranker, and LLM run in-house; on commodity hardware, all three can run on CPU with smaller models. Swapping the LLM requires only a `LLM_BASE_URL` / `LLM_MODEL` environment change — no code changes.

## Limitations

- **Image understanding is textual.** Handover PDFs often contain schematic diagrams critical to troubleshooting. Our current precise-variant pipeline extracts images and can return their URLs for Markdown rendering in Open WebUI (see Section [10](#sec:future), Phase 8a), but semantic retrieval is text-only. Full multimodal retrieval is future work.
- **Document-ingestion throughput is not evaluated at scale.** Real plants reference PDFs by URL; the download dominates wall-clock time. Benchmark B operates on the 10-asset scale that matches realistic shop-floor scopes; ingestion-throughput scaling is a separate concern and out of scope for this paper.
- **Benchmark A uses synthetic shells.** While shell *content* is synthetic (derived from IDTA template skeletons), the *structural* workload — the number and relationship topology of events — reflects a realistic large-repository initial load. We consider this adequate for evaluating the ingestion architecture; evaluating graph semantics at 50 k scale would require 50 k real shells, which no public AAS corpus currently provides.

# Future Work {#sec:future}

**Phase 7.5 — Write evaluation.** A quantitative write-quality study once we have a comparable baseline (possibly via controlled comparison against a non-validated write path).

**Phase 8a — Image URLs in Weaviate.** Store extracted schematic URLs in the chunk metadata, returned alongside text. Open WebUI renders Markdown images inline. Minimal cost, large UX win.

**Phase 8b — Multimodal retrieval.** CLIP/SigLIP embeddings, dual text layers (OCR + VLM-generated alt-text), hybrid search over both.

**Phase 9 — GPU/Triton hybrid dispatcher.** Per-page routing with PyMuPDF pre-flight (empty-page detection, vector density, table-quality heuristics) between local fast and GPU-accelerated precise extraction.

**Phase 10 — PDF-to-AAS extraction.** Batch *and* conversational extraction: agent reads a referenced datasheet, extracts values, and populates the corresponding IDTA submodel elements through the write loop. This is where attachment upload earns its complexity.

**Phase 11 — Multimodal AAS media search.** Images and training videos from the shell, keyframe + transcript chunking, cross-modal reranking (e.g. Jina CLIP v2) on H200/Triton.

# Conclusion {#sec:conclusion}

We have presented a hybrid MCP endpoint that combines graph traversal over a Kafka-synchronised AAS knowledge graph with vector search over referenced documents, augmented by IDTA template awareness and a symmetric CRUD interface. The worker-facing use case — *"transport robot in hall 4 shows a red light, what do I check?"* — is answered end-to-end through a phone browser against an entirely self-hostable stack. A v2 ingestion plugin resolves the transaction-commit bottleneck identified in our ETFA 2025 work and enables a 50× jump in repository scale on the same commodity hardware. An ablation evaluation isolates the contribution of reranking, query rewriting, and template awareness; a direct v1-versus-v2 micro-benchmark causally attributes the ingestion speed-up to the architectural changes. All components are released under MIT licence and deployable via Docker Compose.
